- ### 1.1 项目背景与行业现状

  随着现代社会对生态保护意识的增强，动物园的角色已从单纯的娱乐展示场所，转型为集野生动物保护、科学研究、公众教育和休闲娱乐为一体的综合性机构。这种职能的转变给动物园的日常运营管理带来了巨大的挑战。传统的动物园管理模式往往依赖于纸质记录或分散的电子表格，这种方式在面对庞大的动物种群数据、复杂的医疗记录、精细化的饲养方案以及波动的游客流量时，显得效率低下且容易出错 1。

  在学术界和工业界，"动物园管理系统" (Zoo Management System, ZMS) 的概念应运而生。这是一种旨在通过数字化手段，对动物园内的人、财、物、兽进行全方位管理的综合性软件平台。根据现有的研究，一个成熟的 ZMS 不仅需要处理基础的增删改查（CRUD），还必须具备高度的专业性，例如不仅要记录动物的出生日期，还要追踪其谱系（Pedigree）以避免近亲繁殖；不仅要记录饲料消耗，还要进行营养成分分析 2。

  本项目旨在基于学生本学期所掌握的 **SSM (Spring, Spring MVC, MyBatis)** 后端框架技术和 **Vue.js** 前端框架技术，设计并构思一套符合现代行业标准的动物园综合管理系统。该系统将模拟企业级开发的流程，从需求分析、架构设计到数据库建模，力求实现一个高内聚、低耦合、易扩展的解决方案 1。

  ### 1.2 编写目的

  本《软件需求规格说明书》（SRS）旨在详尽地定义动物园综合管理系统的功能需求、非功能需求、系统接口及数据模型。它是项目开发全生命周期的基石，直接服务于以下目标用户：

  - **开发团队（学生本人）：** 明确编码任务，规范代码结构，确保前后端接口的一致性。
  - **项目导师/评审：** 评估项目的技术深度、业务逻辑的复杂度和工作量。
  - **潜在用户（动物园管理员）：** 确认系统功能是否覆盖了实际运营中的痛点，如门票销售、动物医疗档案管理等。

  本报告将超越简单的功能列表，深入探讨每个模块背后的业务逻辑和数据关联，特别是如何利用 SSM 和 Vue 的技术特性来解决实际问题。例如，如何利用 Spring 的事务管理保证票务库存的一致性，如何利用 Vue 的组件化思想构建可复用的动物信息卡片等。

  ### 1.4 技术栈选择与论证

  本项目采用前后端分离的架构模式。

  - **后端：SSM 框架组合**
    - **Spring:** 作为核心容器，利用 IOC（控制反转）和 AOP（面向切面编程）管理对象生命周期和事务。Spring 的声明式事务管理对于涉及金钱（票务）和库存（饲料）的操作至关重要。
    - **Spring MVC:** 处理 HTTP 请求，遵循 RESTful 风格设计 API 接口。它将请求路由到相应的 Controller，实现了模型、视图和控制器的解耦。
    - **MyBatis:** 作为持久层框架。相较于 Hibernate 全自动 ORM，MyBatis 允许开发者编写原生 SQL。在动物园管理中，经常涉及复杂的多表联查（例如：查询某特定物种在过去一年内患某种疾病的所有个体及其父母的健康状况），MyBatis 提供了更灵活的 SQL 优化空间 4。
  - **前端：Vue.js 生态系统**
    - **Vue.js (v2/v3):** 利用其响应式数据绑定和组件化开发模式。动物园管理后台通常包含大量的数据表格和表单，Vue 的双向绑定能极大提高开发效率。
    - **Vue Router:** 管理单页面应用（SPA）的路由跳转，实现无刷新的页面切换体验。
    - **Vuex / Pinia:** 进行全局状态管理，例如存储当前登录用户的权限 Token，确保不同角色的管理员只能访问授权的模块 16。
    - **Element UI:** 使用成熟的 UI 组件库，快速构建美观、统一的后台管理界面。

  ## 2. 系统架构设计与项目结构

  ### 2.1 总体架构模式

  系统遵循典型的分层架构设计，自下而上分别为：数据持久层（MySQL + MyBatis）、业务逻辑层（Spring Service）、控制交互层（Spring MVC Controller）和前端展示层（Vue.js View）。各层之间通过定义良好的接口进行通信，严格遵循“高内聚、低耦合”的软件工程原则。

  #### 2.1.1 前后端交互流程

  1. **用户操作：** 用户在 Vue 前端界面点击“查询动物”按钮。
  2. **请求发送：** 前端 Axios 拦截器拦截请求，添加 JWT Token 到 Header 中，并发起异步 HTTP GET 请求到后端 API（如 `/api/animals`）。
  3. **请求处理：** Spring MVC 的 `DispatcherServlet` 接收请求，通过 `HandlerMapping` 找到对应的 `AnimalController`。
  4. **业务逻辑：** `AnimalController` 调用 `AnimalService` 接口。
  5. **数据访问：** `AnimalService` 调用 `AnimalMapper` (MyBatis DAO)，执行 SQL 查询语句。
  6. **结果返回：** MySQL 返回结果集，MyBatis 将其映射为 Java POJO 对象，Service 层将其封装为统一的 `Result` 对象（包含状态码、消息、数据）。
  7. **数据渲染：** Vue 前端接收 JSON 数据，更新响应式变量，DOM 自动重绘展示数据列表 18。

  深度解析 - 统一异常处理设计：

  在 common 包中，我们将定义一个全局异常处理器 GlobalExceptionHandler。这是 SSM 开发中的关键组件。当后端发生运行时异常（如数据库连接失败、空指针）时，不能直接将堆栈信息返回给前端，而是通过 @ControllerAdvice 捕获异常，并返回一个格式标准的 JSON 错误信息。这体现了系统的健壮性设计 18。

  Java

  ```
  @ControllerAdvice
  public class GlobalExceptionHandler {
      @ExceptionHandler(Exception.class)
      @ResponseBody
      public Result handleException(Exception e) {
          return new Result(false, StatusCode.ERROR, "系统内部错误：" + e.getMessage());
      }
  }
  ```

  ## 3. 数据库详细设计与建模

  数据库是信息系统的核心。针对动物园管理的复杂性，我们需要设计高度规范化（3NF）的数据库模式，同时在某些报表查询上做适当的反范式处理以优化性能。

  ### 3.1 核心实体关系图 (ERD) 分析

  系统主要包含以下实体及其关系：

  1. **Animal (动物):** 多对一 **Species (物种)**，多对一 **Enclosure (场馆)**。
  2. **MedicalRecord (病历):** 多对一 **Animal**，多对一 **Staff (兽医)**。
  3. **TicketOrder (订单):** 一对多 **TicketItem (票据详情)**，多对一 **Visitor (游客)**。
  4. **Staff (员工):** 多对多 **Role (角色)**。

  ### 3.2 数据表结构详解

  #### 3.2.1 动物基础信息表 (`t_animal`)

  该表存储动物个体的核心数据，是系统的基石 6。

  ## 4. 功能需求详解

  ### 4.1 模块一：动物种群与谱系管理 (Collection Management)

  这是系统的核心业务模块，直接关系到动物园的资产管理和科研价值。

  - **功能点 1.1：动物入园登记 (Accessioning)**
    - **描述：** 管理员录入新动物信息。来源包括：野外救助、园内繁殖、园际交换。
    - **业务规则：** 若来源为“园内繁殖”，系统必须强制要求选择父母（Sire/Dam）。如果父母信息未知，必须标记为“野外捕获”或“未知谱系” 28。
    - **数据验证：** 系统需自动校验 `chip_code` 的唯一性，防止重复录入。
  - **功能点 1.2：谱系与近亲系数分析**
    - **描述：** 系统根据父母ID自动生成家谱树状图。
    - **高级需求：** 集成近亲系数算法（Wright's Coefficient of Inbreeding）。当管理员试图制定繁殖计划（即选择一公一母进行配对）时，系统应自动计算这对组合的潜在后代近亲系数。如果系数超过阈值（如 0.125），系统应弹出红色警告，提示存在遗传病风险 2。这是 ZIMS 区别于普通库存系统的智能化体现。
  - **功能点 1.3：场馆调拨 (Transfers)**
    - **描述：** 记录动物在不同场馆间的移动历史。
    - **场景：** 动物生病需要从“狮虎山”移动到“兽医院隔离区”。管理员发起调拨申请，需经过区域主管审批。系统将自动记录调拨时间、原因、经手人，并更新动物的 `enclosure_id`。

  ### 4.2 模块二：兽医与健康档案 (Veterinary & Health)

  该模块旨在实现医疗流程的无纸化，确保信息的准确传递。

  - **功能点 2.1：电子病历系统 (EMR)**
    - **描述：** 支持兽医录入 SOAP 格式（主观、客观、评估、计划）的诊疗记录。
    - **需求细节：** 界面需支持上传 X 光片、B 超图像等多媒体附件。附件存储在服务器文件系统或 OSS 对象存储中，数据库仅保存文件路径 7。
  - **功能点 2.2：预防医学提醒**
    - **描述：** 系统根据物种的预设模板，自动生成疫苗接种、驱虫和体检的日程提醒。
    - **逻辑：** 设定规则“猫科动物每三个月需进行一次体内驱虫”。系统后台定时任务（Spring Task）每天扫描数据库，若发现某只老虎距离上次驱虫已过 85 天，则向负责兽医发送系统通知或短信提醒 9。
  - **功能点 2.3：检疫隔离管理**
    - **描述：** 对于新入园或患传染病的动物，系统需将其状态锁定为“隔离中”。
    - **约束：** 处于“隔离中”状态的动物，严禁被分配到对公展示区，且禁止发起繁殖计划 33。

  ### 4.3 模块三：饲养与营养管理 (Nutrition & Commissary)

  饲料成本通常是动物园最大的运营支出之一，科学管理至关重要。

  - **功能点 3.1：膳食配方设计 (Diet Formulation)**
    - **描述：** 营养师为特定动物或物种制定每日食谱（如：牛肉 5kg，钙粉 10g，维他命 2片）。
    - **深度洞察：** 系统应关联一个基础营养数据库（如 USDA Nutrient Database），自动计算该食谱的能量、蛋白质、脂肪比例，并与该物种的标准营养需求进行对比，生成雷达图分析 3。
  - **功能点 3.2：库存联动**
    - **描述：** 每日饲养员确认投喂后，系统自动扣减相应的饲料库存。
    - **预警：** 当某种饲料（如冷冻带鱼）库存低于“安全库存线”时，系统自动生成采购申请单，防止断粮风险 35。

  ### 4.4 模块四：票务与游客流量控制 (Ticketing & Visitor Services)

  该模块直接面向公众，对并发性能和用户体验要求极高。

  - **功能点 4.1：动态定价策略 (Dynamic Pricing)**
    - **描述：** 管理员可配置不同日期的票价。
    - **策略：** 支持“早鸟票”、“周末票”、“节假日旺季票”。系统数据库设计需支持价格版本控制，即 `t_ticket_price` 表需包含 `effective_date` 和 `expire_date` 字段，以便系统根据当前日期自动匹配生效价格 36。
  - **功能点 4.2：入园流量控制与限流**
    - **描述：** 根据国家防疫或安全规定，园区每日有最大承载量（如 50,000 人）。
    - **实现机制：** 票务系统需实时统计 `已售出门票数 + 预约年卡入园数`。当达到 95% 阈值时，前端购票页面显示“即将售罄”；达到 100% 时，后端接口拒绝新的下单请求。这需要利用 Redis 原子计数器来保证高并发下的数据准确性，防止超卖 10。
  - **功能点 4.3：验票核销**
    - **描述：** 游客出示二维码，检票员使用手持终端或闸机扫描。
    - **逻辑：** 系统校验二维码有效性（是否过期、是否已使用）。核销成功后，更新订单状态为 `Used`，并记录入园时间戳。

  ### 4.5 模块五：设施维护与工单系统 (Facility Maintenance)

  确保场馆安全是防止动物逃逸和游客受伤的关键。

  - **功能点 5.1：故障报修与工单流转**
    - **描述：** 饲养员发现围栏破损或水泵故障，通过手机端提交报修工单，附带现场照片。
    - **流程：** 提交 -> 维修主管审核派单 -> 维修工接单 -> 维修完成反馈 -> 主管验收归档。整个流程状态流转在数据库中通过 `status` 字段严格控制 12。
  - **功能点 5.2：定期巡检计划**
    - **描述：** 系统自动生成周期性巡检任务（如：每级电网电压检查，每月消防设施检查）。
    - **合规性：** 这些记录是应对上级监管部门检查的重要依据 39。

  ## 5. 用户体验 (UX) 与前端界面设计原则

  考虑到使用人员包括办公室行政（PC端操作为主）和一线饲养员/维修工（移动端操作为主），前端 Vue 项目需采用**响应式布局**。

  ### 5.1 仪表盘设计 (Dashboard)

  管理员登录后的首屏，应提供核心运营指标的可视化概览 40。

  - **实时数据卡片：** 今日入园人数、今日门票收入、在园动物总数、待处理工单数。
  - **图表分析：**
    - 使用 ECharts 绘制“近七日游客流量趋势图”（折线图）。
    - 绘制“动物健康状况分布图”（饼图：健康、生病、隔离）。
    - 绘制“各场馆饲料消耗占比图”（柱状图）。

  ### 5.2 动物档案详情页 (Profile Page)

  这是一个高度集成的信息聚合页面。

  - **头部：** 动物头像、姓名、性别图标、当前状态标签（颜色区分：绿色-健康，红色-生病）。
  - **选项卡 (Tabs) 切换：**
    - **基本信息：** 生日、体重、体长、芯片号。
    - **谱系图：** 嵌入式的小型树状图组件。
    - **医疗记录：** 时间轴（Timeline）形式展示历次看病记录，点击可展开查看详情。
    - **饲养日志：** 表格展示最近 30 天的投喂和排便记录。
    - **位置历史：** 记录该动物历次场馆调动的轨迹。

  ### 5.3 交互反馈设计

  - **加载状态：** 在 Axios 请求过程中，表格区域应显示 Loading 遮罩层，防止用户重复点击。
  - **操作确认：** 涉及删除、状态变更等敏感操作，必须弹出 `MessageBox` 确认框（“您确定要将此动物标记为死亡吗？此操作不可撤销。”）。
  - **表单验证：** 利用 Element UI 的 `el-form` 校验规则，在前端即时反馈错误（如：输入的体重不能为负数，出生日期不能晚于今天）。

  ## 6. 非功能需求 (Non-Functional Requirements)

  ### 6.1 性能要求

  - **响应时间：** 票务查询、动物列表加载等高频操作，API 响应时间应在 200ms 以内。
  - **并发能力：** 票务系统需支持至少 100 QPS (Queries Per Second) 的并发下单请求，确保在节假日高峰期系统不崩溃。
  - **数据库优化：** 对查询频繁的字段（如 `animal_id`, `species_id`, `order_no`）必须建立索引。

  ### 6.2 安全性要求

  - **身份认证：** 基于 JWT (JSON Web Token) 的无状态认证机制。Token 过期时间设为 2 小时，通过 Refresh Token 机制续期。
  - **密码安全：** 用户密码严禁明文存储，必须使用 BCrypt 或 Argon2 算法加盐哈希存储。
  - **API 鉴权：** 后端拦截器需校验用户角色权限，防止越权访问（如：普通饲养员通过 Postman 伪造请求去修改财务数据）。

  ### 6.3 可靠性与数据完整性

  - **事务一致性：** 涉及多表更新的操作（如：动物死亡 -> 更新状态 -> 停止饲料计划 -> 记录尸检报告）必须在同一个 `@Transactional` 事务中，确保要么全成功，要么全回滚。
  - **备份策略：** 数据库需配置每日凌晨自动全量备份，binlog 开启以支持增量恢复。

  ------

  ## 7. 结论与展望

  本需求规格说明书详细阐述了一个基于 **SSM + Vue** 技术栈的现代化动物园综合管理系统的设计构思。该系统不仅仅是简单的增删改查练习，而是深入结合了动物园管理的行业特性——从复杂的谱系追踪到严格的医疗合规，从动态的票务策略到精细的饲料库存。

  通过本项目的实施，开发者将不仅掌握全栈开发的硬技能（Java, SQL, JavaScript），更能领悟企业级应用开发的软技能，如需求分析、领域建模和架构设计。该系统建成后，将为动物园提供一个高效、准确、可视化的管理工具，最终实现提升运营效率、保障动物福利、优化游客体验的三重目标。

  未来，该系统还可进一步扩展，例如接入 IoT 物联网设备（实时监控场馆温湿度）、开发微信小程序端供游客使用导览服务，以及引入 AI 图像识别技术辅助动物个体识别，向着“智慧动物园”的方向不断演进。
